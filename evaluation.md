수업 시간 동안 우리는 gc-board 프로젝트를 통해 Spring Boot의 핵심 아키텍처와 심화 기술을 학습했습니다. 본 수행평가는 이 지식을 바탕으로, 기존 수업 코드에서 발견할 수 있는 '아쉬운 점'을 스스로 진단하고, 이를 개선하거나 확장한 새로운 게시판 서비스를 개발하여 여러분의 설계 능력을 입증하는 데 목적이 있습니다.

> 핵심: 단순히 기능을 구현하는 것을 넘어, "나라면 이렇게 개선했을 것이다"를 명확한 근거와 함께 보여주어야 합니다.

## 1. 🔍 진단 및 개선 (필수 과정)

제시된 gc-board 소스코드는 교육 목적상 간결하게 구성되어 있지만, 실제 서비스 운영 및 유지보수 관점에서는 보완할 부분이 존재합니다. 학생들은 아래 아쉬운 점들의 예시를 참고하여, 새로운 프로젝트에서 개선할 **1~2가지 목표**를 설정하고 이를 명확히 제시해야 합니다.

### 아쉬운 점과 개선 방향 예시

| 아쉬운 점 (예시) | 개선 방향 (학생의 목표)                                                                                   | 적용 기술 (선택 사항)                                                                                       |
| ---------------- | --------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| 단순한 오류 처리 | orElseThrow() 사용으로 인해 예외 발생 시 일관된 HTTP 응답 코드(Global Exception Handler)를 제공하지 못함. | @ControllerAdvice를 활용한 전역 예외 처리기 구현 및 커스텀 비즈니스 예외 정의.                              |
| 코드의 재사용성  | 각 모듈(article, comment)마다 Datasource 설정 등 환경 설정을 반복하고 있음.                               | 외부 설정(Externalized Configuration)이나 공통 모듈을 활용하여 설정의 중앙 관리 및 재사용성 확보.           |
| 테스트 범위      | 신규 기능에 대한 80% 이상의 라인 커버리지 목표만 제시됨.                                                  | 중요 비즈니스 로직(예: 댓글 삭제 로직)에 대한 Mockito 기반 단위 테스트를 100% 확보하고, 테스트 자동화 강조. |

## 2. 🎯 프로젝트 목표 및 범위

새롭게 제작할 게시판은 아래 예시 중 하나를 선택하거나 본인이 자유롭게 정의할 수 있습니다.

### 게시판 예시 (Domain Examples)

- **공지사항 게시판 (Announcement)**: 관리자만 작성 가능, 모든 사용자 조회 가능.
- **Q&A 게시판**: 답변/채택 기능 추가 (댓글 구조 확장).
- **투표 게시판 (Poll)**: 게시글에 투표 기능 추가 (좋아요/싫어요 등 상호작용 확장).
- **이미지/갤러리 게시판 (Image Board)**: 첨부 파일 및 썸네일 기능 추가.
- **가계부/일지 게시판 (Journal)**: 날짜/분류 태그 기능을 통한 검색 및 필터링 강화.

### 필수 구현 기능

| 구분        | 필수 구현 기능                 | 상세 요구사항                                                                                                                |
| ----------- | ------------------------------ | ---------------------------------------------------------------------------------------------------------------------------- |
| 핵심 도메인 | 게시글(Article), 댓글(Comment) | CRUD (생성, 조회, 목록 조회, 수정, 삭제) 기능 완벽 구현.                                                                     |
| 사용자 인증 | 회원가입, 로그인/로그아웃      | JWT, Session 등을 활용하여 로그인 기능을 구현하고, 작성자 본인만 수정 및 삭제 가능하도록 권한 검증(Authorization) 로직 포함. |
| 프론트엔드  | UI/UX                          | 구현한 백엔드 API와 연동하여 모든 기능을 사용할 수 있는 웹 화면을 구성.                                                      |
| 배포        | 라이브 서비스                  | 백엔드 및 프론트엔드를 클라우드 환경에 배포하여 접속 가능한 URL 제공.                                                        |

## 3. ✅ 필수 적용 심화 기술 (최소 1개 이상 선택)

프로젝트에는 아래 기술 목록 중 **최소 1가지 이상**의 심화 기술이 핵심 로직에 적용되어야 합니다. 이는 시연 동영상에서 명확하게 보여줄 수 있는 기술에 중점을 둡니다.

| 번호 | 기술/패턴               | 적용 목표 및 상세 요구사항                                                                                                                                                                                                                                     | 참고 코드 (수업 프로젝트)                                                         |
| ---- | ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| 1    | 커서 기반 무한 스크롤   | 목록 조회 시 LIMIT OFFSET 방식 대신 **lastId**를 활용하여 커서 기반(Cursor-based) 무한 스크롤을 구현해야 합니다. 이는 클라이언트(프론트엔드)에서 무한 스크롤을 요청하는 방식으로 구현되어야 합니다.                                                            | ArticleRepository.findAllInfiniteScroll 및 CommentController.readAll (무한스크롤) |
| 2    | 계층형 댓글 Path Model  | 댓글 기능을 구현할 때, 2Depth 제한이 있는 기존 방식 대신, **경로 문자열(Path Model)**을 사용하여 무한 깊이(N-Depth) 의 계층형 댓글 구조를 구현하고, 계층 구조를 명확히 보여줄 수 있어야 합니다.                                                                | gc.board.comment.entity.CommentPath 및 CommentServiceV2                           |
| 3    | 댓글의 재귀적 삭제 로직 | 댓글 삭제 시, 자식 댓글 유무에 따라 논리적 삭제(Soft Delete) 후, 고아(Orphan)가 된 부모 댓글까지 **재귀적으로 물리적 삭제(Hard Delete)**하는 로직을 구현해야 합니다. 이 로직은 CommentService.delete를 참고하되, 여러분의 프로젝트에 맞게 개선하여 적용합니다. | CommentService.delete(Long commentId)                                             |

## 5. **개발 과정 평가: Git Commit 히스토리 검증**

### **평가의 주요 기준**

본 프로젝트는 **최종 결과물뿐만 아니라, 개발 과정 전체**를 평가합니다. 이를 위해 **Git Repository의 커밋 히스토리**를 검증하여 다음 항목들을 확인합니다:

### **📊 커밋 분산도 (커밋 타이밍)**

- **평가 항목**: 프로젝트 기간 전체에 걸쳐 **균등하게 분산된 커밋**이 있는가?
- **감점 요인**:
  - 하루 또는 이틀에 모든 코드가 몰아서 커밋된 경우
  - 마감일 직전에만 대량의 커밋이 발생한 경우
  - 개발 기간 대부분이 공백인 경우
- **예시**:
  - ❌ **감점 대상**: 12월 17일 하루에만 수십개 이상의 커밋이 집중
  - ✅ **가산점 대상**: 12월 1일 ~ 12월 18일까지 꾸준히 주 3~4회 이상의 커밋 분산

### **🔍 커밋 단위 (기능별 세분화)**

- **평가 항목**: 기능을 **기능별/단위별로 작게 나누어 커밋**했는가?
- **기준**:
  - 한 번의 커밋에 1~3개의 논리적 기능 또는 버그 수정만 포함
  - 관련 없는 여러 기능을 한 번에 커밋하지 않음
  - 테스트 코드, 문서 작성 등도 별도로 커밋
- **예시**:
  - ❌ **감점 대상**: **`git commit -m "전체 기능 완료"`** (한 번에 몇백줄 이상 추가)
  - ✅ **가산점 대상**:
    - **`git commit -m "feat(auth): JWT 토큰 생성 로직 구현"`**
    - **`git commit -m "test(auth): 로그인 단위 테스트 추가"`**
    - **`git commit -m "refactor(article): 리스트 조회 쿼리 최적화"`**

### **📝 커밋 메시지 품질**

- **평가 항목**: 커밋 메시지가 **구체적이고 명확한가?**
- **감점 요인**:
  - "수정", "업데이트", "최종 정리" 같은 모호한 메시지
  - "AI 생성", "ChatGPT 코드" 등 자동 생성 표시
  - 오타나 불완전한 한글/영어 표현

### **📅 커밋 그래프 일관성**

- **평가 항목**: GitHub의 **Contributions 그래프**에서 개발 활동이 일정하게 보이는가?
- **기준**:
  - 주말/주중 구분 없이 꾸준한 커밋 (또는 학사일정에 맞는 합리적 패턴)
  - 공백이 며칠 이상 이어지지 않음
  - 급격한 변동보다는 완만한 곡선

### **💡 학생들을 위한 권장 개발 전략**

1. **계획 수립** (프로젝트 시작 시)
   - 기능을 작은 단위로 분해
   - 각 단위별 예상 작업 시간 계획
   - GitHub Projects 보드에 할일 정리
2. **주기적 커밋** (매일 또는 격일)
   - 하나의 기능이 완성될 때마다 커밋
   - 코드 리뷰/테스트 후 "한 번에" 커밋하지 않기
   - 작은 리팩토링도 별도 커밋
3. **명확한 커밋 메시지 작성**

   # ❌ 나쁜 예

   git commit -m "수정"

   git commit -m "최종"

   # ✅ 좋은 예

   git commit -m "feat(article): 커서 기반 무한 스크롤 구현"

   git commit -m "docs(readme): API 명세 추가"

4. **분산 스케줄링**
   - 일주일에 2~3일은 개발하기
   - 마감 1주일 전 80% 이상 완성
   - 마지막 1주일은 버그 수정 및 배포 준비
