📝 [기획 명세서] 해커톤 통합 운영 플랫폼: Localhost1. 프로젝트 개요 (Project Overview)서비스명: Localhost (로컬호스트)서비스 정의: 해커톤 참가자들의 소통, 분위기 환기, 운영 효율화를 위한 게이미피케이션 기반의 현장 운영 플랫폼.1.1. 기획 배경 (Background & Objective)문제 의식 (Pain Points):장시간 코딩으로 인해 현장 분위기가 경직되고 정적인 경우가 많음.운영진의 공지사항 전달이 어렵거나, 참가 팀 간의 네트워킹/아이스브레이킹 수단이 부족함.해결 방안 (Solution):소통의 장: 익명성을 배제한 '팀 단위' 소통 채널을 제공하여 건전한 경쟁과 교류 유도.청각적 개입: TTS와 신청곡 기능을 통해 정적인 행사장에 활기를 불어넣음.경제 시스템: 코인, 상점, 경매 등 게이미피케이션 요소를 도입해 참가자들의 지속적인 흥미와 참여 유도.기대 효과: 해커톤의 지루함 방지, 팀 간 상호작용 증대, 운영진의 효율적인 행사 통제.2. 사용자 역할 정의 (User Roles)역할 (Role)설명주요 특징User (Team)해커톤 참가 팀팀 단위 1계정 사용. 게시글 작성, 코인 사용, 상점/경매 참여.Admin해커톤 운영진전체 시스템 관리(계정, 재화, 게시글), 경매 진행 권한.Player (Bot)행사장 스피커용 PCTTS 및 신청곡을 실제로 재생하는 클라이언트 뷰. (사람의 조작 없이 자동/반자동 구동)3. 핵심 기능 요구사항 (Functional Requirements)3.1. 인증 및 계정 (Authentication)사전 지급 방식: 회원가입 절차 없이, Admin이 미리 생성한 Team ID / PW를 각 팀에게 불출.팀 정보: 팀명, 현재 보유 코인.3.2. 게시판 및 소통 (Community)게시글 작성 (Post):텍스트 및 이미지 업로드 가능.특수 옵션 (Checkbox):[TTS 요청]: 체크 시 Player 계정의 TTS 큐(Queue)로 텍스트 전송.[노래 신청]: 체크 시 유튜브 URL/곡명 입력 필드 활성화 $\rightarrow$ Player 계정의 뮤직 큐로 전송.반응 (Reaction):별도의 투표 메뉴 없음.게시글에 **이모지(Emoji)**를 달아 의사표현 (👍, ❤️, 🔥, 😂 등).이모지 개수 집계 가능 (인기글 로직).댓글 (Comment):Depth 제한: 최대 2단계 (댓글 $\rightarrow$ 대댓글)까지만 허용. 3단계(대대댓글) 작성 불가.3.3. 경제 시스템 (Economy)코인 (Coin):팀별 가상 화폐 지갑.Transaction Log: 모든 코인 입출금 내역(지급, 구매, 경매 낙찰 등)은 시간/사유와 함께 DB에 기록되어야 함.상점 (Shop):상품 목록(이름, 가격, 이미지, 재고) 조회.구매 시 코인 차감 및 재고 감소.Admin은 상품 추가/수정/삭제 가능.3.4. 이벤트: 경매 (Auction)프로세스:등록: Admin이 경매 물품과 시작가 설정.시작: Admin이 Start 버튼 클릭 시 입찰 활성화.입찰: User는 현재가 + @ 금액으로 입찰 시도. (보유 코인 > 입찰가 조건 필수).종료: 타이머 종료 또는 Admin 강제 종료 시, 최종 입찰자에게 낙찰 및 코인 차감 처리.실시간성: 입찰 현황(현재 최고가, 최고 입찰팀)이 모든 유저에게 실시간 업데이트되어야 함.3.5. 플레이어 기능 (Media Player)이 기능은 별도의 모니터링 페이지(Dashboard) 형태TTS Player:TTS 요청 들어온 텍스트 리스트업.순차적 자동 읽기 또는 Admin 클릭 시 읽기.기능: 재생, 중단(Panic Button).Music Player:신청곡 리스트(Playlist) 시각화.기능: 재생, 일시정지, 다음 곡(Skip), 볼륨 조절.유튜브 임베드(iframe) 등을 활용하여 영상/소리 송출.3.6. 관리자 기능 (Admin Console)대시보드: 전체 팀 코인 현황, 상점 판매 현황 요약.데이터 제어:팀별 코인 수동 지급/차감 (사유 입력).부적절한 게시글/댓글 삭제 (블라인드 처리).경매 물품 등록 및 진행 제어.4. 데이터 로직 및 규칙 (Business Logic)A. 미디어 큐(Queue) 규칙TTS: 게시글 작성 시 TTS가 체크된 경우, 별도의 TTS_Queue 테이블 혹은 리스트에 저장된다. Player에서 읽기가 완료되면 Is_Played: true 처리된다.Music: Music_Queue는 FIFO(선입선출) 구조를 기본으로 하되, Admin이 순서를 강제로 변경하거나 삭제할 수 있다.B. 댓글 계층 구조 (Hierarchy)DB 저장 시 parent_id를 가진다.parent_id == NULL : 1단계 댓글 (Root Comment)parent_id != NULL : 2단계 대댓글 (Reply)제약사항: 대댓글에는 답글 버튼을 노출하지 않거나, 작성 시도를 에러 처리하여 Depth 3 생성을 방지한다.C. 경매 동시성 제어 (Concurrency)여러 팀이 동시에 입찰 버튼을 누를 경우, 서버 도달 순서를 기준으로 1명만 입찰에 성공해야 한다.
